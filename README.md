# PDF Metadata Extractor for Sign Language Literature

一个基于Python的GUI工具，用于批量处理PDF学术文献，自动提取元数据，并过滤出手语相关的研究文献。支持从多个开源学术数据库获取元数据，包括期卷数信息。

[![Python 3.8+](https://img.shields.io/badge/python-3.8+-blue.svg)](https://www.python.org/downloads/)
[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](https://opensource.org/licenses/MIT)
[![GUI: CustomTkinter](https://img.shields.io/badge/GUI-CustomTkinter-green.svg)](https://github.com/TomSchimansky/CustomTkinter)

## ✨ 主要特性

- **📁 批量PDF处理**：支持选择多个文件夹，递归搜索PDF文件
- **🔍 智能DOI提取**：从PDF文本层自动提取DOI，支持OCR备用方案
- **🌐 多数据源支持**：从多个学术数据库获取元数据
  - **OpenAlex** - 全面的学术元数据
  - **Crossref** - 权威的DOI注册数据
  - **Semantic Scholar** - 学术图谱增强数据
- **🎯 手语文献过滤**：基于关键词自动过滤手语相关文献
- **📊 期卷数提取**：自动提取期刊的卷号、期号信息
- **🔤 OCR支持**：集成Tesseract OCR，处理扫描版PDF
- **📂 智能文件分类**：自动将成功/失败的PDF分类到不同文件夹
- **📈 进度跟踪**：实时显示处理进度和状态
- **💾 结果导出**：导出为CSV格式，支持中文字符
- **📝 任务日志**：自动生成处理日志，便于追踪

## 🚀 Alpha 高性能版本 (batch-alpha.py)

Alpha版本针对大规模PDF处理进行了全面优化，适合处理数千篇文献。

### 启动方式
```bash
python batch-alpha.py
```

### 核心优化：多级分流架构

```
┌─────────────────────────────────────────────────────────────┐
│                     PDF 文件输入                              │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  逻辑1: fitz 快速扫描，判断是否有文本层（文字 >= 100 字符）      │
└─────────────────────────────────────────────────────────────┘
                              ↓
         ┌────────────────────┼────────────────────┐
         ↓                    ↓                    ↓
   ┌──────────┐         ┌──────────┐         ┌──────────┐
   │  Tier 1  │         │  Tier 2  │         │  Tier 3  │
   │ (快车道)  │         │ (中车道)  │         │ (慢车道)  │
   ├──────────┤         ├──────────┤         ├──────────┤
   │有文本层   │         │有文本层   │         │扫描件     │
   │+正则DOI  │         │+无DOI    │         │(<100字)  │
   ├──────────┤         ├──────────┤         ├──────────┤
   │正则提取  │         │DeepSeek  │         │OCR提取   │
   │DOI       │         │合并处理   │         │→ Kimi   │
   │          │         │          │         │视觉模型  │
   ├──────────┤         ├──────────┤         ├──────────┤
   │CrossRef  │         │相关性+   │         │页眉页脚  │
   │API验证   │         │元数据    │         │图像提取  │
   └──────────┘         └──────────┘         └──────────┘
```

### 性能对比

| 指标 | 原版 (batch.py) | Alpha版 (batch-alpha.py) |
|------|-----------------|--------------------------|
| 并发处理 | 串行 | 20线程并发 |
| 单文件API调用 | 3-5次 | 1-2次（合并提示词） |
| 超时控制 | 60秒（固定） | 5-120秒（可配置） |
| 5000文件预估 | ~11小时 | ~15分钟 |

### Alpha版新增功能

#### 1. 页面截取（降低内存消耗）
- 处理前截取PDF前N页（默认3页，可配置1-20页）
- 自动创建临时文件，处理完成后自动清理
- 大幅降低大文件处理的内存占用

#### 2. 超时控制（防止卡死）
- 单文件处理超时自动标记为"超时"错误
- 默认20秒，用户可在GUI中配置（5-120秒）
- 5个关键检查点进行超时检测

#### 3. 按错误类型归类
```
归类目录/
├── 完成/                    # 成功文件
│   └── paper1.pdf
└── 失败/                    # 失败文件按错误类型分类
    ├── 超时/
    ├── DOI无效/
    ├── 不相关跳过/
    ├── LLM提取失败/
    ├── 视觉失败/
    └── 处理异常_xxx/
```

#### 4. 实时统计面板
```
📊 Tier1: 120 | Tier2: 350 | Tier3: 80 | ✓成功: 480 | ⏭跳过: 50 | ✗失败: 20
```

#### 5. 完整错误追踪
CSV输出包含 `error_type` 和 `处理车道` 列，便于问题定位。

### GUI配置参数

| 参数 | 默认值 | 范围 | 说明 |
|------|--------|------|------|
| 并发线程数 | 20 | 1-50 | 同时处理的文件数 |
| 超时时间 | 20秒 | 5-120秒 | 单文件处理最大时间 |
| 截取页数 | 3页 | 1-20页 | PDF截取页数，降低内存消耗 |

Alpha版输出包含额外列：

| 列名 | 说明 | 示例 |
|------|------|------|
| **error_type** | 错误类型 | `超时` / `DOI无效` / `不相关跳过` |

## 🐢 Beta 慢速漏斗版本 (batch-beta.py)

Beta版本采用**慢速漏斗策略**，专注于提取准确性而非速度，适合处理高价值PDF文献或对结果质量要求更高的场景。

### 启动方式
```bash
python batch-beta.py
```

### 核心逻辑：四级漏斗策略

Beta版采用四级漏斗策略，逐级尝试提取，确保最大提取成功率：

```
┌─────────────────────────────────────────────────────────────┐
│                     PDF 文件输入                              │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│   第一阶段：首次扫描（前3页）                                  │
│   • 判断相关性（是否为手语文献）                               │
│   • 判断文档类型（期刊/书籍）                                  │
│   • 非相关文献或书籍类型直接跳过                               │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│   第二阶段：DOI提取漏斗                                      │
│   1. 📝 正则表达式提取（优先）                                │
│      └─ 失败 → 2. 👁️ 视觉模型提取（页眉页脚）                  │
│           └─ 失败 → DOI为空，继续处理                         │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│   第三阶段：元数据获取漏斗                                    │
│   1. 🔍 DOI搜索（如果可用）                                   │
│      └─ 失败 → 2. 🔄 复用视觉模型DOI提取结果（如果有）         │
│           └─ 失败 → 3. 🤖 LLM文本提取（完整文本）              │
│                └─ 失败 → 4. 👁️ 视觉模型元数据提取（页眉页脚）  │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│   第四阶段：结果整理与输出                                    │
│   • 格式化作者（APA格式）                                     │
│   • 标准化期卷格式（期(卷)）                                  │
│   • 判断语种（中文/英文）                                     │
└─────────────────────────────────────────────────────────────┘
```

### 漏斗策略特点

#### 1. **跳过优化，专注准确性**
- Beta版不采用Alpha的多级分流架构
- 每个文件都完整执行四级漏斗策略
- 确保每个可能的提取方法都被尝试

#### 2. **视觉模型双重使用**
- DOI提取阶段：视觉模型用于提取页眉页脚的DOI
- 元数据获取阶段：视觉模型再次用于提取完整元数据
- 如果DOI提取阶段的视觉模型已获得元数据，直接复用

#### 3. **LLM文本提取中间层**
- 在DOI搜索和视觉模型之间插入LLM文本提取
- 适用于无法通过API或视觉提取的场景
- 基于完整PDF文本进行分析

### 性能对比

| 指标 | Alpha版 (batch-alpha.py) | Beta版 (batch-beta.py) |
|------|--------------------------|------------------------|
| 处理策略 | 多级分流 | 四级漏斗 |
| 并发处理 | 20线程并发 | 串行处理 |
| 速度 | ⚡️ 极快（约15分钟/5000文件） | 🐢 较慢（约2小时/5000文件） |
| 准确性 | 较高（多级分流） | 极高（逐级漏斗） |
| 适用场景 | 大规模批量处理 | 高价值文献精准提取 |

### Beta版特有功能

#### 1. **文档类型过滤**
- 首次扫描即判断是否为"期刊"类型
- 自动过滤"书籍"类文献
- 非手语相关文献直接跳过

#### 2. **智能页面截取**
- 默认截取PDF前6页（可配置）
- 降低大文件内存占用
- 可完全关闭页面截取功能

#### 3. **串行处理保障**
- 每个文件完整执行漏斗策略
- 避免并发导致的资源竞争
- 适合API调用频率限制严格的情况

#### 4. **多数据源支持**
- 支持OpenAlex、Crossref、Semantic Scholar
- 优先级可配置（默认：OpenAlex → Crossref → Semantic Scholar）

### GUI配置参数

| 参数 | 默认值 | 范围 | 说明 |
|------|--------|------|------|
| 页面截取 | 启用 | 开/关 | 启用页面截取功能 |
| 截取页数 | 6页 | 1-20页 | PDF截取前N页 |


### 何时使用Beta版？

✅ **推荐使用Beta版的情况：**
- 处理高价值、重要文献
- 需要最高提取准确率
- API调用无频率限制
- 处理时间不是首要考量

❌ **推荐使用Alpha版的情况：**
- 处理大量PDF文件
- 需要快速批量处理
- API调用频率受限
- 处理速度至关重要

### 选择建议

**Alpha版 vs Beta版 对比矩阵：**

| 考虑因素 | 推荐版本 | 原因 |
|----------|----------|------|
| 文献数量 > 1000 | Alpha版 | 并发处理大幅缩短时间 |
| 文献重要性高 | Beta版 | 漏斗策略确保准确率 |
| API限制严格 | Beta版 | 串行处理避免频率限制 |
| 处理时间紧迫 | Alpha版 | 多级分流快速完成 |
| 提取失败率高 | Beta版 | 四级漏斗提高成功率 |

> **提示**：对于大多数应用场景，建议先使用Alpha版快速处理，再对失败的文件使用Beta版进行二次尝试。

## 🖥️ 命令行版本 (batch-cli.py)

命令行版本提供无界面、纯命令行的操作方式，适合自动化脚本集成、服务器环境或批量调度任务。

### 启动方式
```bash
python batch-cli.py -i /path/to/pdfs -o results.csv
```

### 核心特性
- **🎯 完全命令行操作**：无需GUI界面，适合自动化
- **⚡️ 高性能处理**：基于Alpha版的多级分流架构
- **📝 丰富参数配置**：支持所有Alpha版功能
- **🔧 灵活输入**：支持文件、文件夹、混合输入
- **📊 详细日志**：可控制日志详细程度
- **🔄 续传支持**：跳过已处理的文件

### 命令行参数详解

#### 基本参数
| 参数 | 缩写 | 说明 | 示例 |
|------|------|------|------|
| `--input` | `-i` | 输入PDF文件或文件夹路径（支持多个） | `-i file1.pdf folder1` |
| `--output` | `-o` | 输出CSV文件路径 | `-o results.csv` |

#### 配置参数
| 参数 | 说明 | 默认值 |
|------|------|--------|
| `--email` | OpenAlex邮箱地址 | `kodwabeamer@gmail.com` |
| `--keywords` | 手语关键词，逗号分隔 | 默认关键词列表 |
| `--sources` | 数据源，逗号分隔 | `crossref,openalex,semantic_scholar` |
| `--workers` | 并发线程数 | `20` |
| `--timeout` | 单文件处理超时时间（秒） | `20` |
| `--pages` | PDF截取页数 | `3` |

#### 功能开关
| 参数 | 说明 |
|------|------|
| `--recursive` | 递归搜索子目录 |
| `--skip-existing` | 跳过输出文件中已存在的记录 |
| `--verbose` | 显示详细日志 |
| `--classify-dir` | 归类目录（成功/失败文件分类存放） |

### 使用示例

#### 示例1：基本使用
```bash
# 处理单个文件夹
python batch-cli.py -i /path/to/papers -o results.csv

# 处理多个文件夹
python batch-cli.py -i /path/to/folder1 /path/to/folder2 -o results.csv

# 处理单个文件
python batch-cli.py -i document.pdf -o results.csv
```

#### 示例2：自定义配置
```bash
python batch-cli.py -i /path/to/pdfs -o results.csv \
  --email your-email@example.com \
  --keywords "sign language, deaf, 手语" \
  --workers 10 \
  --timeout 30 \
  --pages 5 \
  --recursive \
  --verbose
```

#### 示例3：高级功能
```bash
# 启用文件分类
python batch-cli.py -i /path/to/pdfs -o results.csv \
  --classify-dir /path/to/classified

# 跳过已处理文件（续传）
python batch-cli.py -i /path/to/pdfs -o results.csv \
  --skip-existing

# 最小日志输出
python batch-cli.py -i /path/to/pdfs -o results.csv
```

### 故障排除

#### Q: 命令行参数不识别？
**A**: 确保使用正确的参数格式：
- 长参数：`--input` `--output`
- 短参数：`-i` `-o`
- 参数值：`-i value` 或 `-i=value`

#### Q: 中文路径问题？
**A**: 确保系统编码支持UTF-8，或在路径中使用英文

#### Q: 内存不足？
**A**: 减少并发线程数：`--workers 5`

#### Q: API频率限制？
**A**: 增加超时时间：`--timeout 60`

## 🛠️ 安装要求

### 系统要求
- **操作系统**: Windows 10/11, macOS, Linux (已测试)
- **Python版本**: 3.8 或更高版本
- **内存**: 推荐 4GB+
- **磁盘空间**: 100MB 可用空间

### 安装步骤

1. **克隆仓库**
   ```bash
   git clone https://github.com/yourusername/pdf-metadata-extractor.git
   cd pdf-metadata-extractor
   ```

2. **创建虚拟环境 (推荐)**
   ```bash
   python -m venv venv
   # Windows
   venv\Scripts\activate
   # Linux/macOS
   source venv/bin/activate
   ```

3. **安装Python依赖**
   ```bash
   pip install -r requirements.txt
   ```

4. **安装Tesseract OCR (可选，但推荐)**
   - **Windows**: 从 [UB-Mannheim Tesseract](https://github.com/UB-Mannheim/tesseract/wiki) 下载安装
   - **Ubuntu/Debian**: `sudo apt install tesseract-ocr`
   - **macOS**: `brew install tesseract`
   
   安装后，在工具中设置Tesseract路径（默认：`D:\Tesseract\tesseract.exe`）

## 📊 使用CSV合并工具（csv_merge_gui.py）

### 功能介绍
CSV合并工具是一个独立的GUI程序，用于将batch.py输出的CSV文件合并到主表格中，支持自定义处理人和方向信息。

### 主要特性
- **状态过滤**: 自动过滤掉状态为"失败"的记录
- **自定义处理人**: 可手动输入处理人姓名（默认：熊羿成）
- **自定义方向**: 可手动输入研究方向（默认：手语语言学）
- **灵活输出**: 支持覆盖原文件或另存为新文件
- **智能映射**: 自动将batch.py输出格式转换为目标表格格式

### 使用步骤
1. **启动CSV合并工具**
   ```bash
   python csv_merge_gui.py
   ```

2. **选择文件**
   - 点击"选择源文件"：选择batch.py输出的CSV文件
   - 点击"选择目标文件"：选择要追加写入的主表格

3. **配置参数**
   - **处理人**: 输入处理人姓名（默认为"熊羿成"）
   - **方向**: 输入研究方向（默认为"手语语言学"）

4. **选择输出方式**
   - **直接覆盖原目标文件**: 在原文件基础上追加数据
   - **另存为新文件**: 保存为新CSV文件，保留原文件

5. **开始合并**
   - 点击"开始合并"按钮
   - 查看实时进度和日志
   - 完成后会显示合并结果统计

### 列名映射关系
| batch.py输出列名 | 目标表格列名 |
|------------------|-------------|
| `文件名` → `文件名称` |
| `文献名称` → `论文题目` |
| `作者名称` → `作者` |
| `出版年份` → `年份` |
| `发表期刊名称` → `来源（出版社/期刊）` |
| `期卷数` → `期（卷）` |
| `DOI` → `DOI` |
| `error` → `备注` |

### 注意事项
- 程序会自动过滤掉状态为"失败"的记录
- 支持基于"文件名称"列的去重处理
- 保留UTF-8-sig编码，完美支持中文

## ⚠️ 使用时候需要特别注意的地方：

## ⚙️ 配置说明

### 数据源配置
支持以下数据源（按优先级顺序）：
```bash
openalex,crossref,semantic_scholar  # 默认顺序
```

- **openalex**: 全面的元数据，包括概念标签
- **crossref**: 权威的DOI数据，期卷数信息准确
- **semantic_scholar**: 学术图谱数据，补充信息

### 关键词配置
默认关键词用于过滤手语相关文献：
```
sign language, deaf, fingerspelling, asl, csl, gloss recognition
```

可以自定义关键词，用逗号分隔：
```
手语, 聋人, 手势识别, 手语翻译
```

### OCR设置
- **启用Tesseract OCR**: 当PDF文本层不足时自动使用OCR
- **Tesseract路径**: 指向tesseract.exe的完整路径
- **OCR文本阈值**: 少于100字符时触发OCR

## 📊 输出格式

### batch.py输出格式
结果保存为CSV文件，包含以下列：

| 列名 | 说明 | 示例 |
|------|------|------|
| **文件名** | PDF文件名 | `paper123.pdf` |
| **DOI** | 数字对象标识符 | `10.1007/s12345-023-45678-9` |
| **文献名称** | 文章标题 | `A Novel Approach to Sign Language Recognition` |
| **作者名称** | APA格式作者列表 | `Smith, J., & Johnson, A.` |
| **语种** | 文章语言 | `英文` / `中文` |
| **出版年份** | 出版年份 | `2023` |
| **发表期刊名称** | 期刊/会议名称 | `Journal of Deaf Studies` |
| **期卷数** | 格式化期卷数 | `12(3)` |
| **卷** | 卷号 | `12` |
| **期** | 期号 | `3` |
| **数据来源** | 元数据来源 | `openalex` |
| **状态** | 处理状态 | `成功` / `失败` |
| **来源文件夹** | 原始文件夹 | `papers_2023` |
| **error** | 错误信息（如果失败） | `无 DOI` |

### csv_merge_gui.py输出格式
合并后的CSV文件包含以下列（与主表格格式一致）：

| 列名 | 说明 | 示例 |
|------|------|------|
| **文件名称** | PDF文件名 | `paper123.pdf` |
| **论文题目** | 文章标题 | `Sign Language Recognition Study` |
| **语种** | 文章语言 | `中文` / `英语` |
| **作者** | 作者姓名 | `张三, 李四` |
| **年份** | 出版年份 | `2023` |
| **来源（出版社/期刊）** | 期刊名称 | `Journal of Sign Language Studies` |
| **期（卷）** | 期卷信息 | `12(3)` |
| **类型（专著/论文）** | 文献类型 | `论文` |
| **方向** | 研究方向 | `手语语言学`（可自定义） |
| **是否译文** | 是否为译文 | `否` |
| **处理人** | 处理人员 | `熊羿成`（可自定义） |
| **DOI** | 数字对象标识符 | `10.1007/s12345-023-45678-9` |
| **备注** | 错误信息或备注 | `无 DOI` / 空值 |

## 🗂️ 文件分类

```
归类结果目录/
├── 完成/          # 成功处理的PDF
│   └── paper1.pdf
└── 失败/          # 处理失败的PDF
    └── paper2.pdf
```

## 🔗 工具链工作流程

整个工具链的工作流程如下：

```
PDF文件夹 → batch.py → CSV文件 → csv_merge_gui.py → 主表格
```

1. **batch.py**: 提取PDF元数据，生成包含状态信息的CSV
2. **csv_merge_gui.py**: 过滤失败记录，转换格式，合并到主表格

## 📝 使用技巧

### 最佳实践
1. **批量处理前**：先处理少量PDF测试流程
2. **关键词设置**：根据研究领域调整过滤关键词
3. **状态过滤**：利用csv_merge_gui.py自动过滤失败记录
4. **自定义信息**：在合并时设置处理人和研究方向

### 常见问题解决
- **权限错误**：确保CSV文件未被其他程序打开
- **OCR失败**：检查Tesseract安装路径是否正确
- **API超时**：减少并发线程数或分批处理
- **中文乱码**：使用UTF-8-sig编码确保兼容性

## 🔧 高级功能

### 续传处理
如果结果文件已存在，工具会：
- 读取已有的处理结果
- 跳过已处理的文件
- 只处理新增的PDF文件

### 并发处理
- 支持多线程API请求
- 可配置最大工作线程数（默认：3）
- 定期保存结果，防止数据丢失

### 任务日志
每次运行生成日志文件 `{结果文件名}_task_log.txt`：
```
[2024-03-15 14:30] 已完成：
  处理目录：/path/to/papers
  总文件：50，成功：42，失败：8
  结果文件：Result_Metadata.csv
------------------------------
```

### 常见问题

**Q: csv_merge_gui.py无法启动？**
A: 确保已安装customtkinter：`pip install customtkinter`

**Q: 合并后的文件没有自定义处理人信息？**
A: 检查是否在GUI中正确输入了处理人和方向信息

**Q: 状态过滤不生效？**
A: 确保源CSV文件中有"状态"列，且状态值为"成功"或"失败"

**Q: 输出文件编码有问题？**
A: 工具默认使用UTF-8-sig编码，确保Excel等软件正确识别
